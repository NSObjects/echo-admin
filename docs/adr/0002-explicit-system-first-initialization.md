# Use Explicit System First Initialization

Echo Admin will use an explicit **System First Initialization** flow for new installations instead of silently creating the foundational administrator setup during normal process startup. Process startup should report whether the system is uninitialized, while the setup flow performs the one-time creation of the foundational permissions, root role, first administrator, and initial settings. Schema migration remains a process-startup responsibility so setup state can be queried before initialization, but startup must not create or repair foundational business data. After initialization completes, foundational role, permission, menu, API, administrator, and settings data must not be automatically repaired on every startup; future baseline changes should use explicit upgrade work instead. A persistent **Installation State** is the source of truth for whether setup is still allowed, and setup may be retried until completion is recorded. Installation state belongs to the setup module's own persistence model, such as a single-purpose `setup_installation_state` table, rather than editable settings data. Missing installation state means the installation is uninitialized; startup must not infer completion from existing administrators, roles, permissions, menus, APIs, settings, or table contents. Before setup completes, only health/info and setup state/submit endpoints should be reachable; normal administration endpoints, including login, should return a clear uninitialized-system response. `GET /api/setup/state` should expose only a boolean `initialized` field so the public unauthenticated setup probe does not leak administrator, schema, seed, or operational details. That response should use a dedicated `ErrSystemUninitialized` application error with conflict semantics, producing HTTP `409 Conflict` and a safe stable message such as `system is not initialized`. This uninitialized guard should be implemented as server middleware injected by `internal/boot` through a small installation-state interface, so `internal/platform/server` stays generic while every protected route is covered consistently. The setup submit endpoint does not require a separate external setup secret; its safety comes from one-time installation state, input validation, and being permanently closed after initialization. Concurrent setup submissions must be resolved by database transaction and uniqueness constraints instead of frontend button state: only one request may complete setup, while later or competing requests receive an already-initialized conflict. The setup workflow belongs in a dedicated setup business module, while `internal/boot` remains the composition root that wires setup to the other foundational modules. Setup must create the foundational catalog, root role with a stable internal code such as `super_admin`, first administrator assigned to that root role, initial settings, and completion state in one database transaction so a failed setup can be retried without partial business data. The setup usecase owns small outbound interfaces for the foundational writes it needs, including a usecase-owned authorization-foundation interface that creates the highest-authority root role and returns only the role identity needed by the administrator creation step; `internal/boot` provides transaction-aware bridge adapters to access, identity, and settings storage. Setup must not import Casbin or write Casbin policy directly. Casbin remains an authorization runtime detail that interprets the persisted RBAC data produced by access, so future RBAC model changes can be contained behind the authorization-foundation adapter and auth enforcement logic. Root authority should be represented by complete explicit RBAC grants rather than new broad `super_admin` bypasses in auth; existing narrow root-role special cases can be handled separately, but setup should not expand them. Later RBAC baseline changes should use an explicit authorization-upgrade path owned by the access module instead of restarting automatic seed repair, so root authority is maintained through intentional initialization and upgrade operations. Setup should use service logs for success or failure diagnostics without recording passwords, and should not reuse authenticated administrator audit logs because no administrator actor exists before initialization. The browser administration UI should check setup state before current-user loading, route uninitialized installations to a standalone `/setup` page before login, treat the uninitialized error as a global fallback that redirects to `/setup` without preserving the blocked business URL as a login redirect, and must not offer a skip or later path into login or administration. The setup form should collect only `username`, `display_name`, optional `email`, `password`, and optional `site_name` with `Echo Admin` as the default; role, permission, menu, API catalog, and root-role identifiers remain internal setup behavior. Setup success should return only the completed setup state, such as `initialized: true`, and send the browser to the login page instead of automatically creating a **Login Session**. The first administrator is supplied through setup input instead of a fixed `admin` account or a config-file bootstrap password, and the old bootstrap-password configuration path should be removed. This keeps server lifecycle, schema migration, and first administrator onboarding separate so startup failures and setup failures are easier to test, diagnose, and recover from.
