# Use Explicit System First Initialization

Echo Admin will use an explicit **System First Initialization** flow for new installations instead of silently creating the foundational administrator setup during normal process startup. Process startup should report whether the system is uninitialized, while the setup flow performs the one-time creation of the foundational permissions, root role, first administrator, and initial settings. Schema migration remains a process-startup responsibility so setup state can be queried before initialization, but startup must not create or repair foundational business data. After initialization completes, foundational role, permission, menu, API, administrator, and settings data must not be automatically repaired on every startup; future baseline changes should use explicit upgrade work instead. A persistent **Installation State** is the source of truth for whether setup is still allowed, and setup may be retried until completion is recorded. Before setup completes, only health/info and setup state/submit endpoints should be reachable; normal administration endpoints, including login, should return a clear uninitialized-system response. The setup submit endpoint does not require a separate external setup secret; its safety comes from one-time installation state, input validation, and being permanently closed after initialization. The setup workflow belongs in a dedicated setup business module, while `internal/boot` remains the composition root that wires setup to the other foundational modules. Setup must create the foundational catalog, root role, first administrator, initial settings, and completion state in one database transaction so a failed setup can be retried without partial business data. The setup usecase owns small outbound interfaces for the foundational writes it needs, and `internal/boot` provides transaction-aware bridge adapters to access, identity, and settings storage. The browser administration UI should include a setup page that routes uninitialized installations into setup before login. The first administrator is supplied through setup input instead of a fixed `admin` account or a config-file bootstrap password, and the old bootstrap-password configuration path should be removed. This keeps server lifecycle, schema migration, and first administrator onboarding separate so startup failures and setup failures are easier to test, diagnose, and recover from.
