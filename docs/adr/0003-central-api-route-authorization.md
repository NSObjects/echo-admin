# Use Central API Route Authorization Middleware

Echo Admin will authorize managed administration APIs through a single boot-installed route middleware instead of per-handler permission checks. The middleware is mounted on the `/api` group before business modules register routes, receives Echo's registered method/path pattern, and asks auth to authorize the current active role against the persisted API catalog and the role's `api_ids`. Browser Login Session authentication and API Token authentication still happen before this step and only place identity on request context. Bootstrap routes required before the API catalog exists, such as setup state, setup submit, and login, are explicitly skipped. Handler-declared permission tokens are removed from request authorization; permission tokens remain Casbin-backed metadata for current-user grants, menus, buttons, and role-editing catalogs. This keeps every new backend API covered by the same route boundary, avoids duplicating permission constants in handlers, and lets module HTTP adapters stay focused on request parsing, usecase invocation, and response mapping.
